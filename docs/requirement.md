# 반도체 시료 생산주문관리 시스템 - 요구사항 정의서

## 1. 배경

가상의 반도체 회사 "S-Semi"는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등에 납품한다.
시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

최근 주문량 급증으로 다음과 같은 문제가 발생:
- 주문 처리 여부 확인 어려움
- 공정 예약 후 완성 시점 파악 불가
- 재고가 충분함에도 불필요한 추가 공정 발생

기존에 엑셀/메모장으로 주문을 관리하여 실수가 잦고, 재고와 공정 현황 파악이 어려운 문제를 해결하기 위해 체계적인 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 2. 역할별 흐름

- **고객**: 시료 요청자. 필요한 시료를 이메일로 요청.
- **주문 담당자**: 주문서 관리. 요청에 맞게 주문서 작성(등록).
- **생산 담당자**: 시료 생산·승인. 개발 시료 등록, 주문 수신 후 승인 또는 거절.

흐름: 고객 → (시료 요청) → 주문 담당자 → (주문서 전달) → 생산 담당자 → (승인/거절) → 주문 담당자

## 3. 시스템 개요

- **생산 라인**: 공장에서 시료 하나를 생산하는 설비 흐름. 하나의 생산 라인은 시료를 하나씩 생산하며, 주문이 들어온 시료에 대해서만 생산.
- **동작 방식**: 콘솔 기반. 담당자가 직접 명령을 입력하여 시료를 등록하고 주문을 처리.

### 주문 처리 흐름
1. 주문 등록 → `RESERVED`
2. 승인 여부 판단
   - 거절 → `REJECTED`
   - 승인 → 재고 확인
     - 재고 충분 → `CONFIRMED` (출고 준비) → 출고 처리 → `RELEASE` (고객 도달)
     - 재고 부족 → `PRODUCING` (생산 요청) → 생산 완료 시 `CONFIRMED`로 전환 → 출고 처리 → `RELEASE`

## 4. 주문 상태 정의

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외 상태, 모니터링에서 제외) |
| PRODUCING | 주문 승인 완료 및 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 및 출고 대기 중 |
| RELEASE | 출고 완료 |

## 5. 기능 명세

### 5.1 메인 메뉴
- 기능별 선택 화면 표시
- 전체 시료에 대한 요약 정보 확인

| 메뉴 | 의미 |
|---|---|
| 시료 관리 | 새로운 시료 등록, 목록 조회, 이름 검색 기능 |
| 주문 (접수/승인/거절) | 고객 주문 접수 및 생산 라인 담당자의 승인·거절 처리 |
| 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 출고 처리 | CONFIRMED 상태 주문에 대해 출고 실행 |
| 생산 라인 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |

### 5.2 시료 관리
시료(Sample)는 시스템의 가장 기본이 되는 단위이며, 고유한 이름과 속성을 가진다. 시스템에 등록된 시료만 주문 가능하다.

| 메뉴 | 의미 |
|---|---|
| 시료 등록 | 새로운 시료를 시스템에 추가. 속성 값: 시료 ID, 이름, 평균 생산시간, 수율 |
| 시료 조회 | 등록된 모든 시료 목록 확인 (현재 재고 수량 포함) |
| 시료 검색 | 이름 등 속성으로 특정 시료 검색 |

- **수율** = 정상적인 시료 수 / 총 생산 시료 수 (예: 100개 생산 중 90개 정상 = 0.9)

### 5.3 시료 주문
고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.

**시료 예약**
- 고객이 원하는 시료와 수량을 주문
- 이 시점 주문 상태는 `RESERVED`

**예약 시 입력 값**
- 시료 ID
- 고객명
- 주문 수량

### 5.4 주문 승인/거절
- **접수된 주문 목록**: RESERVED 상태의 주문 목록 표시
- **주문 승인**: 접수된 특정 주문에 대해 승인
  - 재고 충분 → 즉시 `CONFIRMED` 상태로 전환
  - 재고 부족 → 생산 라인에 자동 등록, 주문 상태를 `PRODUCING`으로 전환
- **주문 거절**: 접수된 특정 주문에 대해 거절, 즉시 `REJECTED` 상태로 전환

### 5.5 모니터링
주문에 따른 시료 재고 정보를 모니터링하며, 담당자가 시스템 상태를 한눈에 파악할 수 있도록 구성.

**주문량 확인**
- 상태별(RESERVED / CONFIRMED / PRODUCING / RELEASE) 목록 확인
- REJECTED는 유효한 주문이 아니므로 모니터링에서 제외

**재고량 확인**
- 각 시료별 현재 재고 수량 확인
- 주문 대비 재고 수량에 따른 상태 표기
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 수량 부족
  - 고갈: 수량이 0인 상태

### 5.6 생산 라인
생산 라인 정보를 표시하며, 주문량 대비 부족분을 수율과 오차를 고려하여 생산한다.

- **실 생산량** = ceil(부족분 / 수율)
- **총 생산 시간** = 평균 생산시간 × 실 생산량
- 생산 완료 시 주문 상태 `PRODUCING` → `CONFIRMED` 전환

**생산 현황 표기**
- 현재 생산 중인 시료 정보 표시 (표기 수준은 자율 결정, 예: 주문 정보, 현재까지의 생산량 등)

**대기 주문 확인**
- 생산 큐(대기열)를 이용하여 생산 대기 목록 출력
- 스케줄링 전략: FIFO (선입선출)

**재고/생산 처리 규칙**
1. 재고 반영은 "끝나는 즉시" 처리한다.
   - 주문 승인 시 기존 재고로 충당 가능한 만큼은 **승인 즉시** 재고에서 차감한다 (생산 완료를 기다리지 않음).
   - 새로 생산된 수량은 **해당 생산 작업이 완료되는 시점**에 한 번에 재고로 반영된다.
2. 반영은 개당이 아니라 **주문(생산 작업) 단위**로 일괄 처리한다.
3. 생산 소요 시간은 현실 시간(wall-clock)을 기준으로 하며, 프로그램을 재시작해도 진행 중이던 생산의 경과 시간은 영속적으로 유지된다.
4. 실 생산량(ceil(부족분/수율))만큼 생산을 진행하면, 수율에 의한 추가 손실 없이 **전량 정상 생산**된 것으로 처리한다 (수율은 필요 생산량 산정에만 사용, 생산 결과에는 재적용하지 않음).
5. 재고 확인 시 생산 큐(다른 주문이 생산 완료 후 채워 넣을 예정 수량)는 고려하지 않는다. 오직 현재 시점의 재고 수치만 사용한다.

**사례1**
- A제품 수율 50%, 재고 0인 상태에서 50개 주문 접수
- 부족분 50 → 실 생산량 = ceil(50 / 0.5) = 100개 생산 진행
- 규칙 4에 따라 100개 전량 정상 생산됨
- 생산 완료 시 재고 반영(0 + 100 = 100), 그 중 50개는 해당 주문 출고 처리에 사용, 나머지 50개는 재고로 잔존
- 이후 동일 제품 50개 주문이 다시 들어오면, 재고 50개로 즉시 충당 가능 (생산 불필요)

**사례2**
- A제품 여유분(재고) 50개, 수율 50%
- 주문2: 100개 주문 접수 → 승인 즉시 기존 재고 50개 전량을 차감(재고 50 → 0)하여 사용, 남은 부족분 50 → 실 생산량 = ceil(50 / 0.5) = 100개 생산 진행 (PRODUCING)
- 주문3: 50개 주문이 주문2의 생산 완료 이전 시점에 접수
  - 이 시점 실제 재고는 이미 0 (주문2가 선점/차감한 상태) — "재고 50개"는 최초 여유분 기준일 뿐, 이미 다른 주문이 써버린 뒤라 실제 가용량이 아님
  - 재고 확인 시 생산 큐(주문2가 생산 완료 후 채울 예정 수량)는 고려하지 않으므로, 주문3도 부족분 50 → 실 생산량 = ceil(50 / 0.5) = 100개를 추가로 생산해야 한다
- 즉, 두 주문의 등록 시각과 생산 완료 시각의 선후관계에 따라 동일한 재고를 놓고 중복으로 생산이 발생할 수 있으며, 이는 의도된 동작이다 (생산 큐를 참조해 회피하지 않음).


### 5.7 출고 처리
- 재고가 충분해진 `CONFIRMED` 주문에 대해 출고 처리
- 특정 주문에 대해 출고 실행 → 주문 상태 `RELEASE`로 전환

## 6. 과제 진행 유의사항

- 안내된 모델만 사용 (Sonnet / Effort: Medium)
- Model 사용 이력이 없거나 Opus 사용 이력이 존재하면 부정행위로 간주 (동일 조건으로 결과물 평가)
- 과제 종료 시 Claude Code 로그아웃 필요 (`/logout`)

## 7. 미션 소개

### 미션 1: PoC(Proof of Concept) 개발
기능 검증 후 프로젝트 구현 시작. 각 PoC는 개별 Repository로 생성하여 제출.

| PoC 항목 | 설명 |
|---|---|
| MVC 스켈레톤 코드 | Model / Controller / View 패키지 구조와 역할 분리 완성 |
| 데이터 영속성 처리 | JSON 파일 방식으로 데이터 저장·불러오는 구조 구현, CRUD 포함 (nlohmann/json 사용) |
| 데이터 모니터링 Tool | 현재 저장된 데이터 상태를 콘솔에서 실시간 조회할 수 있는 관리자 도구 |
| Dummy 데이터 생성 Tool | 테스트를 위한 Dummy Data 생성 도구 (연결된 JSON 데이터 파일에 추가) |

\* 데이터 영속성: application을 다시 실행해도 데이터를 유지할 수 있는 성질

### 미션 2: 프로젝트 개발

1. CLAUDE.md, PRD.md 등의 문서 관리
2. Harness 도입
3. Test
4. CleanCode
5. Commit 이력
