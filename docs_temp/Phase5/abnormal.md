# Phase 5 특이사항 기록 (요구사항/설계 모호성)

Test 에이전트는 Edit/Bash로 프로덕션 코드나 PRD/요구사항 문서를 수정하지 않고 아래 항목을 기록만 한다.
[testcase.md](testcase.md)의 각 케이스 중 "가정/확정 채택"으로 표시한 것들의 근거를 여기 모았다. Phase 1과 달리 이번 항목들은 사용자에게 아직 직접 확인받지 않았다 — Develope 착수 시 아래 잠정안을 기본값으로 구현하되, 최종 확정은 사용자 판단이 필요하다.

## 1. (Low) 재고 == 대기 수요(RESERVED 합)일 때 "여유"/"부족" 경계값

- requirement.md 5.5절: "여유: 주문 대비 재고 충분", "부족: 주문 대비 재고 수량 부족" — "충분"의 경계(`stock == 수요`를 충분으로 볼지)가 명시되어 있지 않다.
- **잠정 채택(TC-P5-013)**: `stock >= pendingDemand`면 여유, `0 < stock < pendingDemand`면 부족으로 확정했다. "대기 수요를 정확히 전량 충당 가능"하면 그 시점에는 부족한 것이 없으므로 여유로 분류하는 것이 자연스럽다는 것이 근거.
- **영향**: `inventoryStatus()`의 `level` 판정 분기(`stock == 0` → 고갈 최우선, 그 외 `stock >= pendingDemand` → 여유, 아니면 부족).
- **우선순위**: Low — 실무적으로 논란의 여지가 적은 경계 해석이라 판단하지만, Develope 구현 전 사용자 확인을 권장.

## 2. (Medium) 메인 메뉴 요약의 "전체 주문 건수"에 REJECTED 포함 여부

- requirement.md 5.5절은 "주문량 확인" 서브메뉴(모니터링 메뉴 안)에서만 "REJECTED는 유효한 주문이 아니므로 모니터링에서 제외"라고 명시한다.
- requirement.md 5.1절/screens.md 메인 메뉴의 "전체 주문 N건" 요약은 **모니터링 메뉴가 아니라 메인 메뉴 상단**에 표시되는 별개 수치이며, REJECTED 포함 여부에 대한 언급이 전혀 없다.
- **잠정 채택(TC-P5-021)**: 메인 메뉴 요약의 `totalOrderCount`는 REJECTED를 포함한 저장소의 전체 주문 레코드 수로 정의했다("전체"라는 표현을 문자 그대로 해석). 모니터링 메뉴 안의 "주문량 확인"(`countAllStatuses`/`ordersByStatus`)은 기존 확정 규칙대로 REJECTED를 제외한다 — 즉 **같은 시스템 안에 REJECTED를 다루는 기준이 두 곳(메인 메뉴 요약 vs 모니터링 서브메뉴)에서 다르게 적용**된다는 뜻이므로, 사용자 확인이 특히 필요하다.
- **대안**: 만약 "전체 주문"도 REJECTED를 제외해야 한다면 `totalOrderCount`를 `countAllStatuses()`의 합으로 재정의하면 되고, TC-P5-021은 값만 `4`로 바뀌면 된다(테스트 구조 자체는 그대로 재사용 가능).
- **우선순위**: Medium — 화면에 노출되는 수치라 사용자가 직접 보는 값이지만, 어느 쪽이든 구현/테스트 변경 폭이 작다.

## 3. (Medium) "생산라인 대기 건수"가 큐 전체 크기인지, "대기 중"(현재 처리 중 제외)만 세는지

- requirement.md에는 "생산라인 대기 건수"에 대한 정의 자체가 없고, screens.md 메인 메뉴 예시 문구("생산라인 8건 대기")와 `[4] 모니터링`/`[5] 생산라인 조회` 화면이 "현재 처리 중 1건"과 "대기 중인 주문 목록"을 구분해서 보여주는 것이 동시에 존재한다.
- **잠정 채택(TC-P5-022)**: 메인 메뉴 요약의 `productionQueueCount`는 큐 전체 크기(`IProductionQueueRepository::findAll().size()`, 현재 처리 중인 head 포함)로 정의했다 — "대기"라는 단어를 "생산 라인에 걸려 있는 전체 건수"라는 넓은 의미로 해석한 것이다.
- **대안**: 만약 "대기"를 "아직 시작되지 않은 것"으로 좁게 해석해야 한다면 `startedAt`이 빈 문자열인 엔트리 수만 세야 한다(그러면 head는 제외). 이 경우 TC-P5-022의 기대값은 3 → 2로 바뀐다.
- **우선순위**: Medium — `[4] 모니터링` 화면의 `PRODUCING N건 <- 생산라인 대기` 주석은 오히려 "PRODUCING 주문 수 = 생산라인 대기 건수"라는 등식을 암시하므로, 넓은 해석(큐 전체 크기)이 이 주석과 더 자연스럽게 맞는다고 판단해 잠정 채택했지만, 최종 확정은 필요.

## 4. (Low) `ordersByStatus(REJECTED)` 호출 시 동작

- REJECTED는 모니터링에서 "제외"라고만 되어 있고, REJECTED로 명시적으로 조회를 시도했을 때 예외를 던질지 빈 목록을 반환할지는 규정이 없다.
- **잠정 채택(TC-P5-002)**: 빈 목록을 반환한다(예외를 던지지 않음) — 상태별 조회 API 자체는 5개 상태 모두를 받아들이되, 결과 필터링에서 REJECTED만 결과가 없는 것으로 처리하는 편이 API를 더 단순하게 유지한다는 판단.
- **영향**: 크지 않음 — 실제 콘솔 메뉴는 REJECTED를 선택지로 아예 노출하지 않을 것이므로 이 경로가 실사용될 일은 거의 없다.
- **우선순위**: Low.

## 5. `countAllStatuses()`가 0건인 상태를 키로 포함하는지 여부 (설계 선택, 확정 불필요)

- TC-P5-004, TC-P5-030에서 "0건이면 키가 없거나 값이 0인 두 경우 모두 허용"하도록 테스트를 관대하게 작성했다. 이는 모호성이라기보다 `MonitorService`의 내부 자료구조 선택(예: `std::map`을 4개 키로 미리 초기화할지, 존재하는 것만 넣을지)의 문제이므로, Develope의 구현 선택으로 남겨둔다 — 어느 쪽으로 구현해도 화면 출력(`RESERVED 0건`처럼 0도 표시)만 지키면 무방하다.
- **우선순위**: 결정 불필요(구현 자유도).

2. (Medium) 3번 — "생산라인 대기 건수"가 큐 전체 크기인지 "미시작"만인지 — 화면 노출 수치라 사용자 확인 권장.

4. (Low) 4번 — `ordersByStatus(REJECTED)` 동작.
