# Phase 1 특이사항 기록 (요구사항/설계 모순, 모호성)

Test 에이전트는 Edit/Bash 권한이 없어 아래 항목을 임의로 해석해 확정하지 않고 기록만 한다.
[testcase.md](testcase.md)의 각 케이스에서 "가정 A/가정 B" 형태로 분기해 둔 것들의 근거를 여기 모은다.
사용자 확인 후 Develope가 하나를 확정하고, 확정 결과를 이 파일과 testcase.md에 반영한다.

## 1. 시료 등록 시 ID 입력 주체(사용자 입력 vs 시스템 자동 채번) — 문서 간 모순

- `docs/requirement.md` 5.2절: "시료 등록 | 새로운 시료를 시스템에 추가. 속성 값: **시료 ID**, 이름, 평균 생산시간, 수율" — ID를 등록 시 입력하는 속성 값 중 하나로 나열.
- `docs/design.md` §8 이슈 반영 표: "ID 채번 방식(접미사 파싱)의 한계 (DummyDataGenerator) → 본 프로젝트도 동일한 `S{n}`/`O{n}` 채번 방식을 기본으로 유지하되, **Repository 구현체가 채번을 전담**해 Service/Domain 계층이 ID 형식을 직접 알 필요 없게 캡슐화" — ID는 시스템(Repository)이 자동 부여하는 것으로 서술.
- 두 서술이 정면으로 배치된다. requirement.md 5.2절의 "속성 값" 나열이 (a) 사용자가 등록 메뉴에서 실제로 입력하는 값 목록을 의미하는지, 아니면 (b) `Sample` 엔티티가 갖는 속성 전체를 설명한 것뿐이고 실제 입력은 이름/평균생산시간/수율만 받는지 불명확하다.
- **영향받는 테스트케이스**: TC-P1-001, TC-P1-002, TC-P1-002b.
- **요청 사항**: 등록 메뉴에서 사용자가 ID를 직접 입력하게 할지, 시스템이 `S{n}`으로 자동 채번할지 확정 필요.

## 2. 등록 시 초기 재고(stock) 입력 여부

- `docs/requirement.md` 5.2절의 등록 입력값 목록에는 "시료 ID, 이름, 평균 생산시간, 수율"만 있고 **재고(stock)는 없다.**
- 그런데 `Model/Sample.h`(Phase 0 이식 코드)는 `stock` 필드를 갖고 있고, PRD.md 2장 도메인 모델 표에도 "현재 재고 수량"이 포함되어 있다.
- 즉 시료 등록 메뉴에서는 재고를 입력받지 않고 항상 0으로 시작한다고 봐야 하는지, 아니면 등록 시 초기 재고도 입력받을 수 있어야 하는지 requirement.md에 명시가 없다.
- design.md §7의 "Dummy 데이터 생성 기능"은 임의의 재고값을 가진 시료를 만들 수 있다고 서술하지만, 이는 **테스트 픽스처 용도**로 정규 등록 메뉴와는 별개다.
- **영향받는 테스트케이스**: TC-P1-001, TC-P1-011.
- **잠정 가정(테스트케이스에 반영)**: 등록 메뉴는 재고를 입력받지 않고 `stock=0`으로 시작한다. 재고는 이후 주문 승인/생산(Phase 3)을 통해서만 증가한다. 이 가정이 맞는지 확인 필요.

## 3. 수율(yield) 0.0 등록 허용 여부와 Phase 3 division-by-zero 리스크

- design.md §4.4: "평균 생산시간/수율의 유효 범위(0 초과, 수율 0~1)" — 문맥상 "0 초과"는 평균 생산시간에 걸리는 조건인지, 수율에도 걸리는 조건인지 문장 구조가 모호하다("0 초과, 수율 0~1"을 하나의 병렬 나열로 읽으면 평균생산시간>0, 수율은 0~1 범위라는 뜻이 되어 수율 0도 허용됨).
- 그러나 requirement.md 5.6절의 실 생산량 공식 `ceil(부족분 / 수율)`은 수율이 0이면 0으로 나누기가 발생해 Phase 3에서 정의되지 않은 동작(예외/무한대)이 된다.
- **영향받는 테스트케이스**: TC-P1-005.
- **요청 사항**: 등록 시점에 수율 0을 막을지(수율 범위를 "0 초과 ~ 1 이하"로 좁힐지), 아니면 등록은 허용하고 Phase 3에서 별도 예외 처리를 할지 확정 필요. Phase 3 진행 전 반드시 결정되어야 하는 항목이라 우선순위 높음으로 표시.

## 4. 이름 검색의 매칭 방식/대소문자 민감도/ID 포함 여부

- requirement.md 5.2절: "시료 검색 | 이름 등 속성으로 특정 시료 검색" — 다음이 모두 불명확하다.
  1. 부분 문자열(포함) 매칭인지 정확 일치인지.
  2. 대소문자를 구분하는지.
  3. "이름 등"이라는 표현이 이름 이외의 속성(ID 등)도 검색 대상에 포함한다는 뜻인지.
- `docs/design.md`에는 검색 알고리즘에 대한 구체적 언급이 없다(§3 패키지 구조에 `SampleService`가 "등록/조회/검색"을 담당한다고만 되어 있음).
- **영향받는 테스트케이스**: TC-P1-021, TC-P1-022, TC-P1-023, TC-P1-025.
- **잠정 제안(확정 아님)**: 부분 일치 + 대소문자 무시 + 이름만 대상(ID 제외, 최소 스코프)으로 우선 구현하고 필요 시 확장. 사용자 확인 필요.

## 5. 시료 이름 유일성 여부

- requirement.md 5.2절: "시료(Sample)는... 고유한 이름과 속성을 가진다"라는 서술이 있어 이름이 유일해야 한다는 뜻으로 읽힐 수도 있지만, "고유한"이 "이름"에 걸리는지 "이름과 속성의 조합"에 걸리는지(즉 시료 자체가 고유한 존재라는 뜻인지) 불명확하다.
- ID는 명확히 유일해야 하는 키(design.md §4.4 Factory 검증 대상)이지만, 이름까지 유일해야 한다는 명시적 규칙은 없다.
- **영향받는 테스트케이스**: TC-P1-009.
- **잠정 가정(테스트케이스에 반영)**: 이름 중복은 허용(서로 다른 ID를 가진 동명 시료가 여러 건 존재 가능). 확정 필요.

## 6. 빈 검색어 처리 방침

- requirement.md에 빈 검색어에 대한 언급이 전혀 없다.
- **영향받는 테스트케이스**: TC-P1-023.
- **잠정 가정**: 빈 검색어는 "검색어를 입력하세요" 안내 후 재입력을 요구(전체 목록을 반환하는 방식과 사용자 의도 파악이 힘들어 후자를 지양 제안). 확정 필요.

## 우선순위 요약 (Develope/사용자 확인 요청 순서 제안)

1. (High) 3번 — 수율 0 처리 방침: Phase 3 계산 공식과 직결되어 늦어도 Phase 3 착수 전 확정 필요.
2. (High) 1번 — ID 입력 주체: Phase 1 등록 메뉴 UI 자체를 좌우.
3. (Medium) 2번 — 초기 재고 입력 여부: 등록 메뉴 입력 필드 수 결정.
4. (Medium) 4번 — 검색 매칭 방식: 검색 메뉴 동작 결정.
5. (Low) 5번, 6번 — 엣지 케이스, 잠정 가정으로 우선 진행 가능.
