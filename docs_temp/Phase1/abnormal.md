# Phase 1 특이사항 기록 (요구사항/설계 모순, 모호성) [확정 반영본]

Test 에이전트는 Edit/Bash로 프로덕션 코드나 PRD/요구사항 문서를 수정하지 않고 아래 항목을 기록만 한다.
[testcase.md](testcase.md)의 각 케이스에서 애초에 "가정 A/가정 B" 형태로 분기해 두었던 것들의 근거를 여기 모았다.
아래 1~3번은 사용자가 직접 확정했고, 4~6번은 문서에 적혀 있던 "잠정 제안"을 그대로 확정 채택했다(더 이상 미확정 상태가 아니다).

## 1. 시료 등록 시 ID 입력 주체(사용자 입력 vs 시스템 자동 채번) — 문서 간 모순

- `docs/requirement.md` 5.2절: "시료 등록 | 새로운 시료를 시스템에 추가. 속성 값: **시료 ID**, 이름, 평균 생산시간, 수율" — ID를 등록 시 입력하는 속성 값 중 하나로 나열.
- `docs/design.md` §8 이슈 반영 표: "ID 채번 방식(접미사 파싱)의 한계 (DummyDataGenerator) → 본 프로젝트도 동일한 `S{n}`/`O{n}` 채번 방식을 기본으로 유지하되, **Repository 구현체가 채번을 전담**해 Service/Domain 계층이 ID 형식을 직접 알 필요 없게 캡슐화" — ID는 시스템(Repository)이 자동 부여하는 것으로 서술.
- 두 서술이 정면으로 배치되어 사용자에게 확인을 요청했다.

### 확정 결과
- **시스템 자동 채번(`S{n}`, 예: `S1`, `S2`, ...) 채택.** 사용자는 등록 메뉴에서 이름/평균생산시간/수율만 입력하고, ID는 등록 완료 후 시스템이 부여해 화면에 표시한다.
- requirement.md 5.2절의 "속성 값" 나열은 `Sample` 엔티티가 갖는 속성 전체를 설명한 것으로 해석하고(사용자가 실제로 입력하는 값 목록이 아님), 실제 입력은 이름/평균생산시간/수율로 확정한다.
- **영향받는 테스트케이스**: TC-P1-001(ID 자동 채번만 반영), TC-P1-002(결번 처리 — 사용자 ID 입력에 의한 중복 시나리오는 자동 채번 채택 시 발생하지 않음), TC-P1-002b(채번 로직 자체의 검증으로 대체).
- **추가로 발견된 부수 불일치(2차)**: `docs/design.md` §5는 화면 예시(`docs/screens.md`) 근거로 ID 포맷을 `S-{3자리 순번}`(예: `S-001`)로 제시하는데, 이번에 사용자가 확정한 포맷은 `S{n}`(예: `S1`)이다. 두 포맷이 다르다. **Test 역할은 design.md를 직접 수정할 권한/도구가 없으므로**(Edit 없음, 그리고 design.md 수정이 프로덕션 코드/PRD 수정 금지 범위 밖이라 하더라도 이번 작업 범위에는 포함되지 않음), 이 불일치를 기록만 하고 판단은 Develope/사용자에게 넘긴다. testcase.md와 Tests/*.cpp는 사용자가 명시한 `S{n}` 포맷을 기준으로 작성했다. Develope가 실제 구현 시 어느 포맷을 쓸지 최종 확정하고, 필요하면 design.md §5도 함께 갱신해야 한다.

## 2. 등록 시 초기 재고(stock) 입력 여부

- `docs/requirement.md` 5.2절의 등록 입력값 목록에는 "시료 ID, 이름, 평균 생산시간, 수율"만 있고 **재고(stock)는 없다.**
- `Model/Sample.h`(Phase 0 이식 코드)는 `stock` 필드를 갖고, PRD.md 2장 도메인 모델 표에도 "현재 재고 수량"이 포함되어 있다.
- design.md §7의 "Dummy 데이터 생성 기능"은 임의의 재고값을 가진 시료를 만들 수 있다고 서술하지만, 이는 테스트 픽스처 용도로 정규 등록 메뉴와는 별개다.

### 확정 결과
- **등록 메뉴는 재고를 입력받지 않는다.** 등록 시 `stock`은 항상 0으로 초기화되며, 재고는 이후 Phase 3(주문 승인/생산 완료)을 통해서만 증가한다.
- **영향받는 테스트케이스**: TC-P1-001(stock=0 확정 서술), TC-P1-011(재고 5인 케이스는 Phase 3 반영 후 상태를 테스트 픽스처로 직접 시딩해 검증하도록 명시), TC-P1-012(등록 직후 상태만으로 stock=0 재현 가능하도록 갱신).

## 3. 수율(yield) 0.0 등록 허용 여부와 Phase 3 division-by-zero 리스크

- design.md §4.4: "평균 생산시간/수율의 유효 범위(0 초과, 수율 0~1)" — "0 초과"가 평균 생산시간에만 걸리는지 수율에도 걸리는지 문장 구조가 모호했다.
- requirement.md 5.6절의 실 생산량 공식 `ceil(부족분 / 수율)`은 수율이 0이면 0으로 나누기가 발생해 Phase 3에서 미정의 동작이 된다.

### 확정 결과
- **수율 0은 등록 시점에 거부한다(가정 B 채택).** 유효 수율 범위는 **`(0, 1]`**(0 초과, 1 이하)로 확정한다.
- **영향받는 테스트케이스**: TC-P1-005(등록 거부로 변경), TC-P1-006(1.0 허용, 상한 경계), TC-P1-007(범위 밖 값에 0.0도 포함해 재확인).
- Phase 3 착수 전 확정이 필요했던 High 우선순위 항목이었으며, 이번에 해소되어 Phase 3 실생산량 계산 공식(`ceil(부족분/yield)`)이 division-by-zero 걱정 없이 그대로 사용 가능해졌다.

## 4. 이름 검색의 매칭 방식/대소문자 민감도/ID 포함 여부

- requirement.md 5.2절: "시료 검색 | 이름 등 속성으로 특정 시료 검색" — 부분/정확 일치 여부, 대소문자 구분 여부, "이름 등"의 범위(ID 포함 여부)가 모두 불명확했다.
- design.md에는 검색 알고리즘에 대한 구체적 언급이 없다.

### 확정 결과
- **부분 일치(contains) + 대소문자 무시(case-insensitive) + 이름(name)만 검색 대상**(ID 등 다른 속성은 검색 대상에서 제외)으로 확정한다. 애초 잠정 제안 그대로 채택했다.
- **영향받는 테스트케이스**: TC-P1-021(부분 일치 확정), TC-P1-022(대소문자 무시 확정), TC-P1-025(ID 미포함 확정 — "S1" 검색어로는 결과 없음).

## 5. 시료 이름 유일성 여부

- requirement.md 5.2절: "시료(Sample)는... 고유한 이름과 속성을 가진다"라는 서술에서 "고유한"이 이름에 걸리는지, 이름+속성 조합(즉 시료 자체의 유일성)에 걸리는지 불명확했다.
- ID는 명확히 유일해야 하는 키이지만, 이름까지 유일해야 한다는 명시적 규칙은 없었다.

### 확정 결과
- **이름 중복을 허용한다.** 서로 다른 ID를 가진 동명 시료가 여러 건 존재할 수 있다. 애초 잠정 가정 그대로 채택했다.
- **영향받는 테스트케이스**: TC-P1-009(중복 등록 허용으로 확정, 이후 이름 검색 시 두 건 모두 조회됨을 함께 확인).

## 6. 빈 검색어 처리 방침

- requirement.md에 빈 검색어에 대한 언급이 전혀 없었다.

### 확정 결과
- **빈 검색어는 "검색어를 입력하세요" 안내 후 재입력을 요구한다**(검색을 실행하지 않고, 전체 목록을 반환하는 방식은 채택하지 않는다). 애초 잠정 가정 그대로 채택했다.
- **영향받는 테스트케이스**: TC-P1-023(재입력 요구로 확정).

## 잔여 메모 (이번 확정 작업 중 새로 발견된 사항)

- **ID 포맷 이중 기준 문제(위 1번 항목의 부수 불일치)**: `design.md` §5(`S-001` 포맷 예시)와 사용자가 확정한 `S{n}` 포맷이 다르다. Develope 구현 착수 전에 반드시 하나로 정리해야 하며, 정리 결과에 따라 `design.md` §5 예시 JSON도 갱신이 필요할 수 있다. Test 역할은 이를 확정할 권한이 없어 기록만 남긴다.
- **`ISampleRepository`에 채번 전용 메서드가 없음**: 현재 인터페이스는 `findById`/`findAll`/`save`만 있다. `S{n}` 자동 채번을 어느 계층(Repository vs Factory/Service가 `findAll()` 결과로 직접 계산)에서 수행할지는 Develope의 구현 선택으로 남겨둔다. testcase.md TC-P1-002b, `Tests/SampleFactoryTest.cpp`는 "결과적으로 다음 ID가 올바르게 계산된다"만 검증하므로 어느 방식이든 무방하다.

## 우선순위 요약 (해소 상태)

1. ~~(High) 3번 — 수율 0 처리 방침~~ → **확정 완료**(`(0, 1]`).
2. ~~(High) 1번 — ID 입력 주체~~ → **확정 완료**(`S{n}` 자동 채번). 단, ID 포맷 이중 기준 문제는 잔여 메모로 남음.
3. ~~(Medium) 2번 — 초기 재고 입력 여부~~ → **확정 완료**(입력 안 받음, stock=0 시작).
4. ~~(Medium) 4번 — 검색 매칭 방식~~ → **확정 완료**(부분 일치 + 대소문자 무시 + 이름만).
5. ~~(Low) 5번, 6번~~ → **확정 완료**(이름 중복 허용, 빈 검색어 재입력 요구).

모든 항목이 확정되어 이번 Phase 1에서는 더 이상 사용자 확인이 필요한 미결 분기가 없다. 잔여 메모(ID 포맷 이중 기준)만 Develope 착수 전 정리 필요.
